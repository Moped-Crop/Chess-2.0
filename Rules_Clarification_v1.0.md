# Chess 2 — Rules Clarification v1.0

> Название игры: **Chess 2** (внутренний кодовый код дизайна — ASCENT).

> Назначение: закрыть блокеры B1–B8 из Pre-Code Audit окончательными решениями. Без альтернатив, без вариантов на выбор. Каждое правило сформулировано так, чтобы два независимых программиста реализовали его идентично. Этот документ имеет приоритет над предварительными формулировками в GDD §3.6, §5.1, §5.2.
>
> Глоссарий примитивов (используется во всём документе):
> - **LEAPER(offsets)** — для каждого смещения (df,dr): если целевая клетка на доске — можно ходить (если пусто) или бить (если враг). Прыжок игнорирует промежуточные клетки.
> - **SLIDER(dirs)** — для каждого направления (df,dr): шагаем, пока на доске; пустая → ход; первый враг → взятие и стоп; свой → стоп (без хода на неё).
> - **df** = изменение файла, **dr** = изменение ранга. Проверка «на доске» ВСЕГДА по (file,rank), не по сырому индексу (см. B8).
> - **forward** (вперёд) = направление стороны: белые dr=+1, чёрные dr=−1.

## B1. Шах на ходу эволюции

### Окончательное правило
Специального ограничения на шах при эволюции **нет**. Ход, при котором фигура эволюционирует, подчиняется только **общему правилу легальности**: ход легален тогда и только тогда, когда после его применения король ходящей стороны не находится под боем. Эволюционирующий ход **имеет право давать шах** сопернику (и базовым движением, и новыми возможностями формы).

(Это окончательно отменяет предварительную заметку GDD §5.1 «эволюция не может давать шах в свой ход».)

### Примеры
- **Легально:** белая Ладья по файлу f идёт на f7 (в зоне), эволюционирует в Таран и одновременно даёт шах чёрному королю на f8. Ход легален.
- **Легально:** белый Конь идёт на e6 (в зоне), эволюционирует в Ловчего; новый диагональный шаг создаёт шах королю на d5. Ход легален.
- **Легально:** Слон идёт в зону, эволюционирует в Прелата, шаха нет. Легально.
- **Нелегально:** любой ход (в т.ч. эволюционирующий), после которого СОБСТВЕННЫЙ король ходящей стороны оказывается под боем (например, фигура была связана). Нелегально — по общему правилу, не из-за эволюции.

### Implementation Notes
```
legalMoves(state) = pseudoLegalMoves(state)            // включает варианты с эволюцией
    .filter(m => { s2 = applyMove(state, m);
                   return !isKingAttacked(s2, state.turn); })
```
Никаких дополнительных предикатов, связанных с эволюцией и шахом, НЕ добавлять. Шах сопернику не влияет на легальность.

## B2. Точные движения всех 7 эволюционных форм

Все формы определены через примитивы. «Атаки» = множество клеток, которые форма может бить (используется для детекции шаха и контроля полей). Если не сказано иначе, move-клетки = клетки атак (как у обычных фигур). Где есть «move-only», такие клетки в атаки НЕ входят.

Базовые offset-наборы:
```
KNIGHT = {(±1,±2),(±2,±1)}            // 8 смещений
WAZIR  = {(±1,0),(0,±1)}              // 4 ортогональных шага
FERZ   = {(±1,±1)}                    // 4 диагональных шага
ALFIL  = {(±2,±2)}                    // 4 диагональных прыжка на 2
DABBABA= {(±2,0),(0,±2)}             // 4 ортогональных прыжка на 2
DIAGONALS    = {(±1,±1)} как направления слайдера
ORTHOGONALS  = {(±1,0),(0,±1)} как направления слайдера
```

| Форма (RU / код) | База | Движение и взятие | Атаки |
|---|---|---|---|
| **Дозорный / N_OUTRIDER** | Конь | LEAPER(KNIGHT ∪ WAZIR) | те же клетки |
| **Ловчий / N_HUNTER** | Конь | LEAPER(KNIGHT ∪ FERZ) | те же клетки |
| **Прелат / B_PRELATE** | Слон | SLIDER(DIAGONALS) + LEAPER(WAZIR) | те же клетки |
| **Ревнитель / B_ZEALOT** | Слон | SLIDER(DIAGONALS) + LEAPER(ALFIL) | те же клетки |
| **Таран / R_RAM** | Ладья | SLIDER(ORTHOGONALS) + LEAPER(DABBABA) | те же клетки |
| **Опора / R_ANCHOR** | Ладья | SLIDER(ORTHOGONALS) [ход+взятие] + LEAPER_MOVE_ONLY(FERZ) [только ход на пустую, без взятия] | **только** SLIDER(ORTHOGONALS) (диагональные шаги в атаки НЕ входят) |
| **Феникс / ROO_PHOENIX** | Петух | Ход (на пустую): LEAPER_MOVE_ONLY(WAZIR) — 1 шаг в любую из 4 ортогональных сторон. Взятие: forward 1 прямо + 2 диагонали-вперёд на 1 | forward 1 прямо + 2 диагонали-вперёд на 1 |

Пояснения, исключающие разночтения:
- **ALFIL/DABBABA — прыжки:** промежуточная клетка игнорируется (можно перепрыгнуть свою или чужую фигуру), результат зависит только от целевой клетки (пусто → ход, враг → взятие, свой → нельзя).
- **Опора (R_ANCHOR):** диагональный шаг — строго НЕ-взятие (только на пустую) и НЕ создаёт атаки/шаха. Король соперника, стоящий на диагонали-рядом с Опорой, под шахом НЕ находится.
- **Феникс:** «вперёд» — по цвету владельца. forward-прямой: 1 клетка (ход на пустую ИЛИ взятие врага). Две диагонали-вперёд: только взятие. Назад/вбок: только ход на пустую (LEAPER_MOVE_ONLY(WAZIR) включает forward, поэтому forward доступен и для хода). Назад и вбок — НЕ атакуют.

### Implementation Notes
```
function genForm(form, sq, board):
  switch form:
    N_OUTRIDER: return leaper(sq, KNIGHT ∪ WAZIR, board)
    N_HUNTER:   return leaper(sq, KNIGHT ∪ FERZ, board)
    B_PRELATE:  return slider(sq, DIAGONALS, board)   + leaper(sq, WAZIR, board)
    B_ZEALOT:   return slider(sq, DIAGONALS, board)   + leaper(sq, ALFIL, board)
    R_RAM:      return slider(sq, ORTHOGONALS, board) + leaper(sq, DABBABA, board)
    R_ANCHOR:   return slider(sq, ORTHOGONALS, board) + leaperMoveOnly(sq, FERZ, board)
    ROO_PHOENIX:return phoenixMoves(sq, board)

function attacksForm(form, sq, board):   // для isKingAttacked / контроля
    как genForm, НО:
      R_ANCHOR: только slider(sq, ORTHOGONALS, board)          // без FERZ
      ROO_PHOENIX: {forward1 (capture-cell), 2×diag-forward}   // без backward/side
      остальные формы: атаки = их move-клетки
```
`leaperMoveOnly` возвращает только пустые целевые клетки и НЕ участвует в attacks.

## B3. Атаки Петуха (базовая фигура)

### Окончательное определение
Пусть направление вперёд = по цвету владельца (включая Петуха, полученного превращением — наследует направление владельца).

Петух **АТАКУЕТ** (т.е. может взять / держит под боем) следующие клетки:
1. **Вперёд по вертикали:** первую занятую клетку вдоль forward-луча (как ладья атакует вдоль линии). Все пустые клетки до первой занятой — также «контролируются» для целей безопасности короля (король соперника не может встать на forward-луч Петуха при отсутствии преграды между ними).
2. **Две диагонали-вперёд на 1 клетку** (forward-left и forward-right) — всегда (вне зависимости от занятости).

Петух **НЕ атакует**: клетки вбок (боковой шаг — только не-взятие), клетки назад, диагонали-назад.

Король соперника находится под шахом от Петуха ⇔ он стоит на первой занятой клетке forward-луча (без преград между) **или** на одной из двух диагоналей-вперёд.

### Implementation Notes
```
function roosterAttacks(sq, color, board):
  out = {}
  dr = (color == WHITE) ? +1 : -1
  // forward ray: до первой занятой включительно
  step from sq by (df=0, dr):
     while onBoard(next):
        out.add(next)
        if board[next] != empty: break    // первая занятая — атакуется, дальше стоп
        next += (0,dr)
  // два спура
  for df in {-1,+1}:
     t = sq + (df, dr); if onBoard(t): out.add(t)
  return out      // боковые и назад НЕ добавляются
```
Замечание: для генерации ХОДОВ Петуха forward-луч даёт ходы на пустые клетки и взятие первой вражеской; боковой шаг (±1,0) — только на пустую; спуры (±1,dr) — только взятие. Для ШАХА используется `roosterAttacks`, где forward-луч включает первую занятую клетку (это и есть «бьёт короля впереди, если между ними пусто»).

## B4. Превращение и эволюция — полная последовательность

### Окончательное правило
Превращение пешки и эволюция **взаимно исключаются** и **не стэкаются**. Свежепревращённая фигура получает флаг `evolved = true` и **никогда** не эволюционирует по E1.

Порядок обработки внутри `applyMove(state, move)` строго такой:
1. Снять с доски взятую фигуру (если `move.capture` задан; учитывает взятие на проходе).
2. Переместить ходящую фигуру с `from` на `to`.
3. **Превращение:** если ходящая фигура — Пешка и `to` на последнем ряду владельца (белые rank_index 7 / чёрные 0): заменить её на `move.promotion` (Q/R/B/N/ROO), установить `evolved = true`. **E1 на этом ходу НЕ выполняется** (даже если `to` лежит в зоне эволюции).
4. **Иначе — Эволюция (E1):** если ходящая фигура — рабочая (N, B, R, ROO) И `evolved == false` И `to` лежит в её зоне эволюции (см. B8): заменить её на форму:
   - N/B/R → `move.evolveTo` (одна из двух допустимых форм данной базы);
   - ROO → ROO_PHOENIX;
   установить `evolved = true`.
5. Иначе — обычный ход (трансформации нет).
6. Обновить права рокировки, поле взятия на проходе, счётчик (B6), номер хода; переключить очередь.

Дополнительно:
- Король и Ферзь не являются рабочими фигурами → E1 к ним неприменим никогда.
- Фигура с `evolved == true` (эволюционировавшая ИЛИ полученная превращением) по E1 не трансформируется повторно.
- Превращение пешки **обязательно** (как в шахматах); генератор обязан выдавать 5 вариантов `promotion`.
- Для рабочей фигуры, входящей в зону с выбором формы (N/B/R), генератор обязан выдавать **по одному ходу на каждую допустимую форму** (2 варианта); для ROO — один (Phoenix).

### Пример
- Пешка f7→f8: только превращение (выбор из 5), `evolved=true`, E1 не срабатывает, хотя f8 в зоне.
- Конь d4→e6 (в зоне, `evolved=false`): эволюция; `move.evolveTo ∈ {N_OUTRIDER, N_HUNTER}`; на доске появляется выбранная форма.
- Таран (уже `evolved=true`) ходит e6→e7 внутри зоны: трансформации нет.

### Implementation Notes
```
applyMove(state, move):
  s = clone(state)
  if move.capture != null: s.board[move.capture] = empty
  p = s.board[move.from]; s.board[move.from] = empty
  if p.type == PAWN and isLastRank(move.to, p.color):
        p = { type: move.promotion, color: p.color, hasMoved:true, evolved:true }
  else if isWorking(p.type) and !p.evolved and inEvoZone(move.to, p):
        p = { type: evolvedTypeOf(p, move), color: p.color, hasMoved:true, evolved:true }
  else:
        p.hasMoved = true
  s.board[move.to] = p
  updateCastling(s, move); updateEnPassant(s, move, p)
  updateHalfmove(s, move, evolvedThisMove)   // см. B6
  s.turn = opposite(s.turn); if s.turn==WHITE: s.fullmove++
  return s

isWorking(t) = t in {N, B, R, ROO}          // базовые рабочие типы
```
`evolvedThisMove` истинно, если на шаге 4 произошла замена на форму.

## B5. Бастион (рокировка) — точные поля назначения

### Окончательные поля
Старт: Король f1 (белые) / f8 (чёрные); Ладьи a1,j1 / a8,j8.

| Сторона | Кто | Из | В |
|---|---|---|---|
| Белые, к j (kingside) | Король | f1 | **h1** |
| | Ладья | j1 | **g1** |
| Белые, к a (queenside) | Король | f1 | **d1** |
| | Ладья | a1 | **e1** |
| Чёрные, к j (kingside) | Король | f8 | **h8** |
| | Ладья | j8 | **g8** |
| Чёрные, к a (queenside) | Король | f8 | **d8** |
| | Ладья | a8 | **e8** |

Король всегда смещается ровно на 2 клетки к выбранной Ладье; Ладья встаёт на клетку, через которую прошёл Король (рядом с ним со стороны центра).

### Условия легальности (все обязаны выполняться)
1. У Короля и выбранной Ладьи `hasMoved == false` (отслеживается флагами прав рокировки).
2. Все клетки строго между Королём и выбранной Ладьёй пусты:
   - kingside (f↔j): g1, h1, i1;
   - queenside (f↔a): e1, d1, c1, b1.
   (Учесть: Петух на d1/g1 — между Королём и Ладьёй; рокировка возможна только после его ухода.)
3. Король не под шахом в исходной позиции.
4. Король не проходит через битую клетку и не встаёт под бой. Проверяемые клетки:
   - kingside: f1, g1, h1 — ни одна не должна быть атакована соперником;
   - queenside: f1, e1, d1 — ни одна не должна быть атакована соперником.
5. Ход кодируется как `special = 'castle-king' | 'castle-queen'`.

После хода: Король и Ладья получают `hasMoved = true` (соответствующие права рокировки снимаются).

Индексы (для самопроверки, B8): f1=5,g1=6,h1=7,i1=8,j1=9; e1=4,d1=3,c1=2,b1=1,a1=0. Kingside: K 5→7, R 9→6. Queenside: K 5→3, R 0→4. (Чёрные: +70 к индексам.)

### Implementation Notes
```
castleKing(color): K: f→h (+2 файла), R(j): →g
castleQueen(color): K: f→d (−2 файла), R(a): →e
between_kingside = [g,h,i]; pass_kingside = [f,g,h]
between_queenside = [e,d,c,b]; pass_queenside = [f,e,d]
legal iff: rights ok && between all empty && none of pass_* attacked && king not in check
apply: move K and R to target squares; clear castling rights for that king
```

## B6. 75-ходовое правило — сброс счётчика

### Окончательное правило
Счётчик «полуходов без прогресса» (`halfmoveClock`) ведётся в ПОЛУХОДАХ.
- **Сбрасывается в 0** при любом из событий на текущем полуходу:
  1. взятие любой фигуры (включая взятие на проходе);
  2. ход пешкой (одинарный, двойной, взятие на проходе, превращение);
  3. **эволюция** рабочей фигуры по E1 (`evolvedThisMove == true`).
- **Иначе** увеличивается на 1.
- Ничья фиксируется, когда `halfmoveClock` достигает **150** (= 75 полных ходов обеими сторонами без прогресса).

(Порог 150 — текущее значение конфигурации; правило СБРОСА окончательно и от значения порога не зависит.)

### Implementation Notes
```
updateHalfmove(s, move, evolvedThisMove):
  if move.capture != null
     || movingPieceWasPawn(move)
     || evolvedThisMove:
        s.halfmoveClock = 0
  else:
        s.halfmoveClock += 1
// терминал:
if s.halfmoveClock >= 150: result = DRAW
```

## B7. Повторение позиции — идентичность

### Окончательное определение
Две позиции идентичны (для троекратного повторения) тогда и только тогда, когда совпадают ВСЕ компоненты:
1. **Содержимое всех 80 клеток**, где «фигура» = пара {тип, цвет}. Эволюционированная форма — это ОТДЕЛЬНЫЙ тип (например, R ≠ R_ANCHOR), базовый Петух ≠ ROO_PHOENIX. Флаг `hasMoved` сам по себе в идентичность НЕ входит (его влияние учитывается через права рокировки, п.3).
2. **Очередь хода** (кто ходит).
3. **Права рокировки** — какие из 4 рокировок (белые-K, белые-Q, чёрные-K, чёрные-Q) ещё доступны.
4. **Поле взятия на проходе** — значение `enPassant` (индекс клетки 0–79 или «нет»).

В идентичность НЕ входят: `halfmoveClock`, номер хода.

Троекратное повторение = позиция с одинаковым ключом встретилась 3 раза за партию (не обязательно подряд) → ничья.

Замечание: т.к. эволюция необратима и меняет тип фигуры, позиция «после эволюции» никогда не совпадёт с позицией «до эволюции» — ложных повторений не возникает.

### Implementation Notes
```
positionKey(state):
  cells = for s in 0..79: encode(board[s])      // 0=empty, иначе уникальный код {type,color}
  return join(cells) + "|" + turn
       + "|" + castlingBits(state)              // 4 бита: WK,WQ,BK,BQ
       + "|" + (enPassant==null ? "-" : enPassant)
// учёт:
counts[positionKey(state)] += 1
if counts[key] >= 3: result = DRAW
```
`encode` обязан давать разные коды базовым и эволюционным типам и разным цветам.

## B8. Координатная система — полная формализация

### Окончательное определение
- Доска: 10 файлов × 8 рангов = 80 клеток.
- Файлы a..j → file_index 0..9. Ранги 1..8 → rank_index 0..7.
- Индекс клетки: `s = rank_index * 10 + file_index`, диапазон 0..79.
  - a1 = 0, j1 = 9, a8 = 70, j8 = 79.
  - `file_index(s) = s % 10`; `rank_index(s) = s div 10` (целочисленно).
- Белые внизу: домашний ряд rank_index 0, пешки rank_index 1. Чёрные вверху: домашний ряд rank_index 7, пешки rank_index 6.
- Направление «вперёд»: белые dr=+1 (Δindex +10), чёрные dr=−1 (Δindex −10).
- Проверка нахождения на доске и шага ВСЕГДА через (file,rank), не через сырой индекс (защита от перехода через край):
  - целевая клетка от `s` со смещением (df,dr) валидна ⇔ `0 ≤ file_index(s)+df ≤ 9` и `0 ≤ rank_index(s)+dr ≤ 7`.
- Последний ряд (превращение): белые rank_index 7; чёрные rank_index 0.
- **Зоны эволюции** (по rank_index):
  - Рабочие N/B/R: белые {5,6,7} (ранги 6–8); чёрные {0,1,2} (ранги 1–3).
  - Петух (ROO): белые {6,7} (ранги 7–8); чёрные {0,1} (ранги 1–2).

### Implementation Notes
```
fileOf(s) = s % 10
rankOf(s) = s div 10
sq(file,rank) = rank*10 + file
onBoard(file,rank) = 0<=file<=9 && 0<=rank<=7

inEvoZone(to, piece):
  r = rankOf(to)
  if piece.type == ROO:
     return piece.color==WHITE ? (r in {6,7}) : (r in {0,1})
  else: // N,B,R
     return piece.color==WHITE ? (r in {5,6,7}) : (r in {0,1,2})

isLastRank(to, color) = color==WHITE ? rankOf(to)==7 : rankOf(to)==0
```
ВСЯ геометрия ходов/слайдеров/прыжков обязана использовать `onBoard(file,rank)` по (df,dr), а не арифметику по сырому индексу.

## Итог

### Список принятых решений
- **B1.** Ограничения «шах на эволюции» нет; легальность — только по безопасности своего короля. Эволюция может давать шах.
- **B2.** 7 форм заданы через примитивы: Дозорный=N+W, Ловчий=N+F, Прелат=B+W, Ревнитель=B+Alfil, Таран=R+Dabbaba, Опора=R + ход-FERZ(без взятия/атаки), Феникс=ход-WAZIR + взятие forward1/диагонали-вперёд.
- **B3.** Атаки Петуха = forward-луч до первой занятой включительно + 2 диагонали-вперёд; бок/назад не атакуют; «вперёд» по цвету.
- **B4.** Превращение и эволюция взаимно исключаются; превращённая фигура `evolved=true` и по E1 не растёт; зафиксирован порядок шагов `applyMove`.
- **B5.** Бастион: kingside K f→h, R j→g; queenside K f→d, R a→e (и зеркально для чёрных); заданы условия и проверяемые клетки.
- **B6.** `halfmoveClock` сбрасывается на взятии, ходе пешки и эволюции; порог ничьей 150 полуходов.
- **B7.** Идентичность позиции = расстановка(тип с учётом формы + цвет) + очередь + права рокировки + поле e.p.; повтор 3× = ничья.
- **B8.** Индекс = rank*10+file, a1=0; «вперёд» по цвету; зоны эволюции и последний ряд заданы по rank_index; геометрия — только через (file,rank).

### Обновлённая готовность к кодингу
**~90% → достаточно для старта.** Все правила, влияющие на генерацию ходов, легальность, шах/мат/пат, ничьи, спецходы и эволюцию, формализованы однозначно. Оставшиеся ~10% — исключительно **числовые балансовые параметры** (номиналы фигур, тонкая настройка форм, граница зоны как балансовый выбор, точный порог хода-лимита, таблица «достаточного материала», тайм-пресеты). Они НЕ влияют на корректность движка и настраиваются на плейтесте.

### Подтверждение
После принятия B1–B8 **игровой движок можно реализовывать без дополнительных дизайнерских решений.** Любой оставшийся открытый пункт относится к балансу/числам, а не к правилам, и имеет безопасные дефолты, уже зафиксированные в этом документе и в Tech Plan v1. Спецификация однозначна: два независимых программиста, следуя B1–B8 и B-примитивам, получат идентичное поведение движка.
