# Chess 2 — Технический план реализации v1

> Название игры: **Chess 2** (внутренний кодовый код дизайна — ASCENT).

**Цель документа:** достаточная техническая база для старта разработки браузерной версии (MVP: локальная игра на одном устройстве, без сервера).
**Источник правил:** раздел 0 GDD («Каноническая спецификация MVP»). Этот документ описывает КАК, GDD — ЧТО.

Принцип №1: **чистое детерминированное ядро правил, полностью отделённое от UI.** Ядро не знает про DOM/React, тестируется само по себе и переиспользуется (будущие ИИ/онлайн).

---

## 1. Стек технологий

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | **TypeScript** (strict) | Типобезопасность критична для движка правил; меньше класс ошибок |
| Сборка/дев | **Vite** | Быстрый dev-server, простой билд, первоклассная поддержка TS |
| UI-фреймворк | **React 18** | Меню/панели/модалки; знакомый, стабильный |
| Рендер доски | **SVG** (React-компоненты) | 10×8 — лёгкая нагрузка; чёткость, простой хит-тест, масштабируемость, доступность. Canvas — запасной вариант, если понадобится тяжёлая анимация (не ожидается) |
| Состояние UI | **Zustand** (лёгкий store) | Простое управление состоянием партии/таймеров без бойлерплейта Redux |
| Тесты | **Vitest** + **@testing-library/react** | Юнит-тесты движка (вкл. perft) и компонентов |
| Качество | **ESLint + Prettier**, **tsc --noEmit** в CI | Единый стиль, статическая проверка |
| Хранение | **localStorage** (autosave/настройки) + **IndexedDB** (именованные сейвы, опц.) | Локально, без сервера |
| Упаковка | **PWA** (vite-plugin-pwa) | Оффлайн-игра, установка на устройство |
| i18n | **простой словарь строк** (RU/EN) | Локализация без тяжёлых зависимостей |

Зависимости намеренно минимальны. Никаких игровых движков (Phaser/Unity) — игре нужна логика правил, а не игровой цикл реального времени.

## 3. Архитектура фронтенда

### 3.1 Поток данных (однонаправленный)
```
Пользователь → UI-событие → store.action → engine (чистая функция) → новое GameState → ререндер
```
- UI **никогда** не мутирует состояние напрямую и не реализует правила — он только вызывает движок и рисует результат.
- Store хранит: текущий `GameState`, историю состояний (undo/redo), выбранную клетку, набор легальных ходов для выбранной фигуры, состояние таймеров, настройки.

### 3.2 Рендер доски (SVG)
- Сетка 10×8 как `<g>` клеток; фигуры — отдельный слой `<g>` поверх; слой подсказок (легальные ходы, последний ход, шах, зона эволюции) — между ними.
- Хит-тест по координате клетки (data-атрибуты или вычисление по позиции).
- Ориентация: матрица трансформации для разворота доски (для hotseat — опц. разворот к активной стороне).
- Анимация перемещения — CSS transform/transition или Web Animations API; короткая (~150–200 мс). Эволюция — отдельная подсветка.

### 3.3 Ввод хода
1. Тап/клик по своей фигуре → store запрашивает у движка легальные ходы этой фигуры → подсветка.
2. Тап по цели из набора → если ход ведёт к эволюции с выбором (Конь/Слон/Ладья) → открыть `EvolutionModal` → выбранная форма кладётся в объект `Move`.
3. `store.applyMove(move)` → движок возвращает новое состояние → ререндер, переключение часов, запись в историю.
- Drag-and-drop — альтернативный путь к тому же `applyMove`.
- Нелегальные ходы невозможны (UI предлагает только легальные).

### 3.4 Адаптивность/доступность
- Доска масштабируется по контейнеру (aspect-ratio 10:8); панели сворачиваются на узких экранах.
- Темы/палитры через CSS-переменные; дальтоник-режимы; опц. координаты; клавиатурная навигация (стрелки + Enter) — базово в MVP, расширенный screen-reader — пост-MVP (отметить).

## 4. Архитектура игрового движка

### 4.1 Базовые типы (эскиз)
```ts
type Color = 'white' | 'black';
type PieceType = 'K'|'Q'|'R'|'B'|'N'|'P'|'ROO'           // базовые (ROO = Петух)
  | 'N_OUTRIDER'|'N_HUNTER'|'B_PRELATE'|'B_ZEALOT'         // эволюции коня/слона
  | 'R_RAM'|'R_ANCHOR'|'ROO_PHOENIX';                      // эволюции ладьи/петуха
interface Piece { type: PieceType; color: Color; hasMoved: boolean; evolved: boolean; }
type Square = number;                  // 0..79, index = rank*10 + file (10 файлов, 8 рядов)
interface Move {
  from: Square; to: Square;
  capture?: Square;                    // включая взятие на проходе
  promotion?: PieceType;               // превращение пешки
  evolveTo?: PieceType;                // выбранная форма при эволюции
  special?: 'castle-king'|'castle-queen'|'enpassant';
}
interface GameState {
  board: (Piece|null)[];               // длина 80
  turn: Color;
  castling: { /* флаги по ладьям/королю */ };
  enPassant: Square | null;
  halfmoveClock: number;               // для лимита ходов
  fullmove: number;
  history: string[];                   // хеши позиций (троекратное повторение)
  result: 'ongoing'|'white'|'black'|'draw';
}
```
**Представление доски:** массив 80 клеток (mailbox), индекс `rank*10+file`. Для проверки границ — либо явные проверки file/rank, либо mailbox с рамкой. Bitboard'ы не нужны (нет требований к перфомансу движка-ИИ в MVP). **[Решение зафиксировано: mailbox 10×8.]**

### 4.2 Конвейер генерации ходов
```
generatePseudoLegal(state) → [Move]      // по правилам каждой фигуры, без учёта шаха
   → для каждого: applyMove → isKingInCheck(mover)?  // фильтр легальности
   → дополнительно: фильтр «эволюция не даёт шах в свой ход»
   = legalMoves(state)
```
- **Скользящие** (Ладья/Слон/Ферзь/части эволюций/Петух-вперёд): лучи до препятствия.
- **Прыжковые** (Конь): фикс. смещения.
- **Петух:** вперёд-луч (ход и взятие первого врага), боковой шаг на 1 (только ход), диагональ-вперёд на 1 (только взятие). Никаких назад/вбок-взятий.
- **Спецходы:** Бастион и взятие на проходе — в `special.ts`, добавляются в общий список.
- **Эволюция:** не отдельный «ход», а свойство хода: если рабочая фигура заканчивает ход в зоне (Петух — ряды 7–8/1–2; прочие — дальняя треть), генерируются варианты `Move.evolveTo` (две формы для N/B/R → два хода-кандидата; Петух → один). Применяется в `apply.ts`.

### 4.3 Проверка легальности / шах
- `isKingInCheck(state, color)`: генерировать атаки соперника на поле короля (или обратная атака-проверка по типам — быстрее). 
- Ход легален, если после `applyMove` собственный король не под боем.
- Спец-ограничение эволюции (не давать шах в момент превращения) проверяется отдельным предикатом после применения.

### 4.4 Терминальные состояния (`terminal.ts`)
- **Мат:** сторона под шахом и `legalMoves == []`.
- **Пат:** не под шахом и `legalMoves == []` → ничья (MVP). [Флаг конфигурации `stalemate = draw|win` — заложить как параметр правил для будущего Competitive, но в MVP = draw.]
- **Троекратное повторение:** по хешам позиций (учитывать очередь хода, права рокировки, e.p., статусы эволюции фигур).
- **Лимит ходов:** `halfmoveClock >= 75` (параметр) без взятия/хода пешки → ничья. **[Требует плейтеста: значение.]**
- **Недостаток материала:** таблица «материал, неспособный матовать» — **[Требует плейтеста с эволюционными фигурами; в MVP консервативная заглушка: только K vs K и явные случаи].**

### 4.5 Reducer и неизменяемость
- `applyMove(state, move): GameState` — чистая функция, возвращает НОВОЕ состояние (структурное копирование доски). Это даёт бесплатно undo/redo (стек состояний) и безопасные тесты.
- Все правила сосредоточены здесь и в `moveGen` — UI их не дублирует.

### 4.6 Тестируемость (критично)
- **Perft** (подсчёт числа листьев дерева на глубину N из заданных позиций) — золотой стандарт проверки генератора ходов; завести набор эталонных позиций (старт + специально сконструированные для Петуха, эволюции, Бастиона, e.p.).
- Юнит-тесты на каждое правило: ход Петуха во всех направлениях, запрет отступления, триггер зоны, выбор формы, запрет шаха-на-эволюции, рокировка сквозь Петуха (запрещена, пока он не ушёл), превращение, мат/пат, повторение/лимит.
- Тесты сериализации: `state → FEN-ext → state` (round-trip), `replay(moves) == finalState`.

## 5. Система проверки ходов (move validation)

Двухуровневая, единый источник истины — движок:
1. **Генеративная (основная):** UI получает от движка СПИСОК легальных ходов выбранной фигуры и позволяет только их. Нелегальный ход физически нельзя совершить.
2. **Верифицирующая (защита):** перед `applyMove` движок повторно проверяет, что `move ∈ legalMoves(state)` (защита от рассинхрона/багов UI, основа для будущего онлайна с недоверенным клиентом).

Эволюция и спецходы валидируются как часть легальности (форма указана в `Move`, проверяется допустимость: фигура рабочая, поле в зоне, фигура ещё не эволюционировала, ход не даёт шах в момент эволюции).

## 6. Хранение состояния партии

### 6.1 Представление в памяти
- Единый неизменяемый `GameState` (см. 4.1) + стек истории состояний (для undo/redo) и список ходов (для записи/реплея).

### 6.2 Формат сериализации
- **Позиция:** расширенный FEN под 10×8 + Петух + эволюции + флаги (рокировка, e.p., halfmoveClock, очередь). Расширения: новые символы фигур (например, `O` — Петух; формы — двухсимвольные коды в скобках или отдельная карта эволюций), 10 файлов на ряд.
- **Партия (сейв):** JSON
```json
{
  "version": 1,
  "rules": { "boardWidth":10, "stalemate":"draw", "halfmoveLimit":75, "evolutionZone":"third" },
  "startPosition": "<ext-FEN>",
  "moves": ["<coordinate/extended move>", "..."],
  "clocks": { "mode":"5+3", "whiteMs":..., "blackMs":... },
  "result": "ongoing"
}
```
- **Почему через список ходов + стартовая позиция:** компактно и устойчиво (детерминизм гарантирует, что реплей восстанавливает точное состояние). Дополнительно можно кэшировать снапшот текущей позиции для мгновенной загрузки без реплея.

### 6.3 Версионирование
- Поле `version` + `rules` фиксируют правила, по которым шла партия (на случай будущих изменений баланса) — старые сейвы воспроизводимы.

## 7. Сохранения

- **Автосейв текущей партии** в `localStorage` после каждого хода (ключ `ascent.autosave`). Восстановление при перезапуске («Продолжить»).
- **Именованные сохранения:** список в `IndexedDB` (или localStorage для MVP) — сохранить/загрузить/удалить.
- **Экспорт/импорт:** скачивание/загрузка JSON-файла партии (шеринг, бэкап). Также экспорт записи партии в человекочитаемую нотацию (PGN-аналог).
- **Настройки и прогресс обучения** — отдельные ключи в localStorage.
- Надёжность: запись по схеме с `version`; при несовместимости — мягкая деградация/миграция.

## 8. Локальный мультиплеер на одном устройстве (hotseat)

- Два игрока ходят по очереди на одном устройстве. Движок и так очередями управляет; UI лишь отражает активную сторону.
- **Опция разворота доски** к активному игроку (toggle): «авто-разворот», «всегда со стороны белых», «не разворачивать».
- **Часы** переключаются на активную сторону при подтверждении хода (см. 9).
- **Приватность ввода:** не требуется (полная информация — обе стороны всё видят; это шахматы).
- **Undo:** разрешён в casual (по обоюдному, т.к. одно устройство); в режимах с часами/рейтингом — запрещён/ограничен. **[Требует плейтеста: политика отмены.]**
- Архитектурно hotseat — частный случай общего матч-движка; будущий онлайн/ИИ подключаются как альтернативные «контроллеры ходов» без изменения ядра.

## 9. Таймеры

- **Модель данных:** `{ mode, whiteMs, blackMs, incrementMs, delayMs, activeColor, lastTickTs }`. Хранится в store, НЕ в `GameState` ядра (часы — это слой матча, не правил; ядро остаётся чистым и детерминированным).
- **Отсчёт:** `requestAnimationFrame`/таймер обновляет отображение; фактическое списание считается по `performance.now()` (монотонные часы) — устойчиво к фоновым вкладкам/дрейфу.
- **Переключение:** при подтверждении хода — списать прошедшее у активной стороны, применить инкремент/делей, передать ход.
- **Режимы:** none / sudden-death / Fischer (X+Y) / delay. Пресеты: Bullet/Blitz/Rapid/Classical/Custom.
- **Флаг:** активная сторона на 0 → проигрыш, кроме случая недостатка матующего материала у соперника → ничья. **[Требует плейтеста: «достаточный материал» с эволюциями; значения пресетов под 10×8.]**
- **Пауза:** только в casual без часов или по обоюдному (hotseat).

## 10. Порядок разработки по этапам

Каждый этап завершается «definition of done» (DoD) и тестами. Ядро — первым.

**Этап 0 — Каркас (инфраструктура).**
Vite+TS+React, ESLint/Prettier, Vitest, CI (`tsc`, lint, tests). DoD: пустое приложение собирается и деплоится; тест-раннер работает.

**Этап 1 — Ядро правил без эволюции.**
Типы, доска 10×8 (mailbox), генерация ходов классических фигур + Петуха, легальность/шах, мат/пат, базовые ничьи; сериализация (ext-FEN), хеш позиции. DoD: проходят юнит-тесты правил + **perft** на эталонных позициях; партию можно сыграть программно (без UI).

**Этап 2 — Эволюция + спецходы + превращение.**
Триггер зоны, формы (Конь/Слон/Ладья ×2, Петух→Феникс), выбор формы, ограничения (одноразовость, запрет шаха-на-эволюции, без K/Q), Бастион, взятие на проходе, превращение пешки. DoD: тесты на все спецслучаи; perft с этими ходами.

**Этап 3 — Играбельный UI (hotseat).**
SVG-доска, отрисовка фигур/форм, подсветка легальных ходов и зоны, ввод (тап/drag), `EvolutionModal`, история ходов, индикаторы шаха/последнего хода, экран нового матча. DoD: полноценная партия двух людей на одном устройстве от начала до мата/ничьей.

**Этап 4 — Таймеры.**
Модель часов, режимы/пресеты, переключение, флаг, отображение. DoD: корректная игра на время в hotseat, устойчивость к фоновым вкладкам.

**Этап 5 — Сохранения.**
Автосейв/продолжить, именованные сейвы, экспорт/импорт JSON, запись партии (PGN-аналог), undo/redo (casual). DoD: партия сохраняется, восстанавливается реплеем бит-в-бит, импорт/экспорт работают.

**Этап 6 — Обучение и подсказки.**
Интерактивный тур (фигуры → Петух → зона → эволюция → мат), микро-уроки, песочница, базовые детерминированные пазлы; слои подсказок (легальные ходы, индикатор эволюции, предупреждения). DoD: новый игрок проходит тур и играет.

**Этап 7 — Полировка.**
Визуальные темы/наборы фигур, значки эволюции, анимации, звуки, настройки, доступность (контраст/дальтоник/координаты/клавиатура), PWA-оффлайн, i18n RU/EN. DoD: премиальное ощущение; оффлайн-установка.

**Этап 8 — Хардненинг и подготовка к плейтесту.**
Покрытие тестами краевых случаев, профилирование, фикс багов; сборка инструментов для сэлф-плея/балансовых замеров по метрикам (частота эволюций, длительность, доля ничьих); свод всех `[Требует плейтеста]` в трекер. DoD: стабильная альфа, готовая к балансовым итерациям.

**Зависимости:** 1→2 (ядро до эволюции), 1/2→3 (правила до UI), 3→4→5, 3→6, всё→7→8. Этапы 1–2 можно вести параллельно с дизайном ассетов (этап 7).

## 11. Риски реализации и меры

| Риск | Мера |
|---|---|
| Баги генерации ходов (особенно Петух/эволюции) | Perft + обширные юнит-тесты как gate каждого PR |
| «Протечка» правил в UI (дублирование логики) | Архитектурное правило: `engine` не зависит от `app`; UI только вызывает движок |
| Рассинхрон состояния/часов | Часы вне ядра; состояние — immutable + verify-перед-apply |
| Неверифицированные эндшпили/материал | Консервативные заглушки в MVP + пометки `[Требует плейтеста]` |
| Изменения баланса ломают старые сейвы | Версионирование правил в формате сейва |
| Производительность анимаций | SVG + короткие CSS-анимации; Canvas как запас (не ожидается нужным) |

## 12. Что НЕ входит в MVP (явно, чтобы не расползалось)
Онлайн-мультиплеер и матчмейкинг; ИИ-оппонент и подсказка лучшего хода; рейтинги/лиги; режим Competitive с альт-правилами (пат=победа и т.п.); сетап-драфт; расширенные форматы (Pantheon/Dominion); монетизация/косметика-магазин; серверная инфраструктура. Архитектура (чистое ядро + контроллеры ходов) оставляет дверь для всего этого открытой без переписывания.
