## О проекте

**Chess 2** — браузерная шахматная игра-вариант. Отличия от классики:
- Доска **10 файлов × 8 рангов = 80 клеток** (файлы `a..j`, ранги `1..8`).
- Новая фигура **Петух** (`ROO`, Rooster) — ходит/бьёт по-особому (см. B3).
- Механика **эволюции**: рабочая фигура (Конь/Слон/Ладья/Петух), завершая ход в своей «зоне эволюции», превращается в усиленную **форму** (одноразово, необратимо).
- MVP: локальная игра на одном устройстве (hotseat), **без сервера и ИИ**.

Источники правды (НЕ менять без явной просьбы пользователя) — все в `docs/`:
- `docs/Rules_Clarification_v1.0.md` — **окончательные** правила B1–B8. Приоритет над всем остальным.
- `docs/Tech_Plan_v1.md` — техплан (стек, структура, этапы).
- `docs/Pre_Code_Audit.md` — порядок разработки (§7) и что вырезано из MVP (§4).
- `docs/GDD_ASCENT_Standard.md` и прочие `docs/Chess_2_*.md` — фоновый дизайн-контекст.

## Железные правила архитектуры

1. **Чистое ядро.** Папка `engine/` не импортирует НИЧЕГО из `app/`, React или DOM.
   Зависимость только в одну сторону: `app/` → `engine/`.
2. **Движок детерминирован и иммутабелен.** `applyMove(state, move)` возвращает
   НОВОЕ состояние (структурная копия доски), не мутирует старое. Это даёт
   бесплатный undo/redo и безопасные тесты.
3. **UI не реализует правила.** UI спрашивает у движка список легальных ходов
   и показывает только их. Нелегальный ход физически невозможен.
4. **Часы — вне ядра.** Таймеры живут в слое матча (store), не в `GameState`.

Структура `src/`:
```
engine/   types, board, attacks, pieces/, moveGen, apply, legality, terminal, hash, special, evolution, index
app/      store/, components/, persistence/, i18n/, App.tsx, main.tsx
```

## Координатная система (B8) — используется ВЕЗДЕ

- Индекс клетки: `s = rank*10 + file`, диапазон `0..79`.
- `a1 = 0`, `j1 = 9`, `a8 = 70`, `j8 = 79`. **Белые внизу** (домашний ранг 0).
- `fileOf(s) = s % 10`, `rankOf(s) = Math.floor(s / 10)`, `sq(file,rank) = rank*10 + file`.
- «Вперёд»: белые `dr = +1`, чёрные `dr = −1`.
- Проверка «на доске» ВСЕГДА через `(file, rank)`, НИКОГДА по сырому индексу
  (иначе фигура «перепрыгнет» край доски):
  `onBoard(file,rank) = 0<=file<=9 && 0<=rank<=7`.
- Последний ранг (превращение): белые `rank 7`, чёрные `rank 0`.
- Зоны эволюции (по `rankOf`): N/B/R — белые `{5,6,7}`, чёрные `{0,1,2}`;
  Петух (ROO) — белые `{6,7}`, чёрные `{0,1}`.

## Ключевые правила (краткий справочник B1–B8)

- **B1 — шах на эволюции.** Спец-ограничения НЕТ. Ход легален ⇔ после него свой
  король не под боем. Эволюционирующий ход может давать шах сопернику.
- **B2 — 7 форм** (через примитивы LEAPER/SLIDER): Дозорный=Конь+Wazir,
  Ловчий=Конь+Ferz, Прелат=Слон-слайдер+Wazir, Ревнитель=Слон-слайдер+Alfil,
  Таран=Ладья-слайдер+Dabbaba, Опора=Ладья-слайдер + Ferz-ТОЛЬКО-ХОД (без
  взятия/атаки), Феникс=Wazir-ход + взятие forward1/двух диагоналей-вперёд.
- **B3 — атаки Петуха.** Forward-луч до первой занятой включительно + 2 диагонали-
  вперёд. Бок и назад НЕ атакуют. «Вперёд» = по цвету владельца.
- **B4 — превращение ≠ эволюция.** Взаимно исключаются, не стэкаются.
  Свежепревращённая фигура `evolved=true` и по E1 никогда не эволюционирует.
  Порядок в `applyMove`: снять взятие → переместить → превращение ИЛИ эволюция → обычный ход.
- **B5 — Бастион (рокировка).** Старт: Кр `f1/f8`, Ладьи `a/j`. Kingside (к j):
  Кр f→h, Л j→g. Queenside (к a): Кр f→d, Л a→e. Зеркально для чёрных.
- **B6 — счётчик 75 ходов.** `halfmoveClock` (в полуходах) сбрасывается в 0 при:
  взятии, ходе пешкой, ИЛИ эволюции. Иначе +1. Ничья при `>= 150`.
- **B7 — повторение позиции.** Ключ = содержимое 80 клеток {тип(с формой),цвет} +
  очередь + права рокировки + поле e.p. (БЕЗ halfmoveClock и номера хода). 3× = ничья.
- **B8 — координаты** (см. раздел выше).

Типы фигур (`PieceType`): базовые `K Q R B N P ROO`; формы
`N_OUTRIDER N_HUNTER B_PRELATE B_ZEALOT R_RAM R_ANCHOR ROO_PHOENIX`.

## Онлайн-слой (добавлен после MVP)

Проект вырос за пределы MVP: есть аккаунты, друзья и онлайн-партии 1×1.

- `server/` — Express + Socket.IO + PostgreSQL (`pg`, сырой SQL, миграции в
  `server/db/migrations/`). `server/gameEngine.ts` импортирует движок НАПРЯМУЮ
  из `src/engine` — отдельной серверной реализации правил НЕТ и быть не должно.
- Онлайн — **второй контроллер ходов** поверх того же `gameStore`:
  `mode: 'local' | 'online'`; общий путь применения — `applyConfirmedMove`;
  свой ход онлайн идёт через `submitOnlineMove` (оптимистично + socket),
  ход соперника — `applyRemoteMove`. Локальный режим не изменён.
- Безопасность: JWT в httpOnly-cookie, CSRF double-submit, zod-валидация всего
  входа, rate limiting (REST + token-bucket на сокетах), helmet.
- Секреты в `.env` (DATABASE_URL, JWT_SECRET); `NODE_ENV` — в `.env.server`
  (в `.env` его класть НЕЛЬЗЯ — Vite сломает прод-сборку React).
- Тесты сервера и интеграционные — на `pg-mem` (`DATABASE_URL=memory://`
  поддерживается самим сервером); боевую БД тестами не трогать.
- Разработка: `npm run dev:all` (vite :5173 + tsx watch server :3001,
  прокси `/api` и `/socket.io` в vite.config.ts). E2E: `npm run test:e2e`.

## Безопасность аккаунта (фаза 1, добавлено после онлайн-слоя)

Подтверждение почты, восстановление пароля, 2FA, удаление аккаунта. Миграция
`server/db/migrations/002_account_security.sql`.

- **Подтверждение почты — обязательное.** `register` создаёт аккаунт, но НЕ
  сессию (отвечает `{ status: 'verify_email_sent', email }`) и шлёт письмо со
  ссылкой `/verify-email?token=`. `login` при `email_verified = false` отдаёт
  `403 email_not_verified` и не ставит cookie. `POST /api/auth/verify-email`
  подтверждает почту и сразу ставит сессию (мгновенный вход из письма).
  Отсюда **инвариант: любая живая сессия ⇒ почта подтверждена** — отдельной
  проверки `email_verified` на защищённых роутах нет и не нужно.
- **Пользователи до этой миграции** помечены `email_verified = true` (колонка
  добавлена с `DEFAULT true`, затем дефолт переключён на `false` для новых) —
  им ничего подтверждать не пришлось.
- **`token_version`** зашит в JWT (`{ uid, tv }`). `requireAuth(pool, secret)`
  сверяет `tv` с базой на каждый запрос; та же сверка на подключении сокета.
  Смена пароля и удаление увеличивают `token_version` → все прежние сессии на
  всех устройствах разлогиниваются. При смене пароля инициатору сразу ставится
  свежая cookie (остаётся в системе).
- **2FA** — TOTP (`otplib` v13, функциональный API), опционально, с 8
  резервными кодами. Секрет хранится **зашифрованным** (AES-256-GCM,
  `TOTP_ENCRYPTION_KEY`). При включённой 2FA `login` отдаёт короткий
  challenge-JWT (`purpose: 'login2fa'`, 5 мин) **в теле ответа** (не cookie!),
  затем `POST /api/auth/login/verify-2fa` ставит настоящую сессию.
- **Удаление аккаунта — это анонимизация, а не `DELETE`** (`games.white_id/
  black_id` ссылаются на `users(id)` без `ON DELETE CASCADE` — историю партий
  других игроков ломать нельзя). `username → deleted_user_<id>`, `email →
  уникальный placeholder`, секреты обнуляются, `deleted_at = NOW()`. Два вызова
  `POST /api/account/delete`: пароль → код (TOTP или на почту) → подтверждение.
- **Одноразовые токены писем** — в базе только `sha256`-хэш (как CSRF), не сам
  токен (высокая энтропия, bcrypt не нужен).
- **2FA — второй фактор и на смену пароля/логина/почты** (если у пользователя
  включена 2FA): эти эндпоинты в `account.ts` требуют `code` (TOTP/резервный)
  через `verify2faIfEnabled`. Смена display_name (`PUT /api/profile`) и удаление
  аккаунта — как были. После подтверждения почты новичку предлагается сразу
  включить 2FA (`/secure-account`, компонент `TwoFactorEnable`). На экранах
  входа/регистрации есть переключатель языка (RU/EN), и язык письма совпадает с
  языком интерфейса (клиент шлёт `lang`). Пароль на регистрации вводится дважды;
  поля пароля — компонент `PasswordInput` с «показать/скрыть».
- **Новые файлы:** `server/lib/{mailer,totp,crypto,tokens,twofactor}.ts`,
  `server/routes/account.ts`. Почтовик инъектируется в `createApp` (в тестах —
  записывающий фейк из `tests/server/testApp.ts`). Тесты — на `pg-mem`.
  Особенности pg-mem: UNIQUE-индекс не терпит `NULL` (поэтому email при
  удалении — placeholder, а не NULL); конкатенация `text || integer` требует
  явного `::text`.
- **Почта — через HTTP-API Resend, НЕ SMTP.** Railway режет исходящие
  SMTP-порты (25/465/587) на всех тарифах кроме Pro, поэтому `mailer.ts` шлёт
  письма POST-запросом на `api.resend.com/emails` (порт 443). Отправитель —
  адрес на подтверждённом в Resend домене (`chess2-ascent.online`). Домен на
  Cloudflare (NS), сайт — кастомный домен Railway через Cloudflare-проксирование.
- **Новые env-переменные** (все в `REQUIRED`): `RESEND_API_KEY`, `MAIL_FROM`,
  `APP_URL`, `TOTP_ENCRYPTION_KEY` (32 байта hex, обязан совпадать локально и на
  Railway). На Railway также нужен `NPM_CONFIG_PRODUCTION=false` (иначе с
  `NODE_ENV=production` не ставятся devDependencies и сборка падает на `tsc`).

## Онлайн-таймер и история партий (фазы 2+3)

Миграция `server/db/migrations/003_online_timer_and_history.sql`: у games
появились `time_control_id`, `white_ms`, `black_ms`, `turn_started_at`,
`win_reason` + индекс истории.

- **Часы онлайн-партии — сервер авторитетен.** Математика часов НЕ
  дублируется: `server/lib/serverClock.ts` переиспользует чистые функции из
  `src/app/clock/clock.ts` (`presetById`, `applyElapsed`, `switchAfterMove`,
  `flaggedColor`) — тем же приёмом, что `gameEngine.ts` импортирует движок.
  ВАЖНО: на сервере источник времени ВСЕГДА `Date.now()` (переживает
  перезапуск, согласуется с TIMESTAMP), на клиенте — `performance.now()`;
  при приёме серверного снэпшота клиент пересчитывает только `lastTickTs`
  (`whiteMs`/`blackMs` копируются как есть). Не смешивать источники.
- **Контроль времени задаётся при приглашении** (`friend-invite` требует
  `timeControlId`; zod-enum собирается программно из `PRESETS`). Часы
  инициализируются при `invite-accepted`, списываются в том же UPDATE, что
  пишет ход; свежий снэпшот `clock` летит сопернику в каждом событии `move`.
- **Тайм-аут**: `flagTimers` (Map по gameId, зеркалит `abandonTimers`) —
  партия завершается сама, без хода. Два слоя защиты после перезапуска:
  восстановление таймеров при старте `attachGameSockets` + ленивая проверка
  `checkFlagged` в `join-game`. В `finishGame` флажок снимается безусловно.
- **`win_reason` персистится для ВСЕХ причин** (`'game' | 'resign' |
  'abandon' | 'timeout'`, тип EndReason — не голая строка) и отдаётся в
  `game-state` (`reason`) вместе с `clock`/`timeControlId`. Клиентский тикер
  в онлайне исход НЕ решает — при нуле зажимает показ и ждёт серверный
  `game-over`.
- **История — только онлайн-партии** (локальный hotseat на сервер не
  отправляется — осознанно). `server/routes/games.ts`: `GET
  /api/games/history?page=` (по 20, `hasMore`) и `GET /api/games/:id` —
  только участнику, чужим — 404 как несуществующая. Страницы `/history`
  (GameHistoryPage) и `/history/:id` (GameReplayPage — листание кадров,
  клавиши ← →).
- **`mode: 'replay'` в сторе — не путать с `'online'`**: доска не кликается,
  `loadReplayFrame` НЕ пишет автосейв (в отличие от `loadMatch`), выход —
  `exitReplay`. Заход на `/play/online/:id` завершённой партии редиректит на
  `/history/:id` (фикс кнопки «назад»: причина теперь в БД, а не только в
  разовом сокет-событии).
- В онлайн-партии вкладки «Ходы/Настройки»; `OnlineSettingsTab` — только
  язык/тема/звук (общие поля вынесены в `settingsFields.tsx`).

## Багфиксы, брендинг, обучение (фазы 4+5)

- **CSP и аватарка:** `imgSrc` в helmet включает `blob:` — обрезка аватара
  читает выбранный файл через `URL.createObjectURL`; без этого загрузка
  блокировалась для ЛЮБОГО формата (дело было не в HEIC).
- **Регистрация после опечатки в почте:** `register` перед проверкой
  уникальности УДАЛЯЕТ (настоящий DELETE, не анонимизация) строку с тем же
  username/email, если `email_verified = false` и `deleted_at IS NULL` —
  неподтверждённый аккаунт не мог быть «использован» (без сессии в него не
  войти), а анонимизированные не трогаем (на них ссылаются партии).
- **Лоадер маршрутизации** (`router.tsx`) — отложенный: первые 300 мс ничего
  не рендерит (иначе мигал при быстрой проверке сессии). Логика
  RequireAuth/HomeRedirect не менялась.
- **Анимация перемотки повтора:** построение `LastAction` вынесено в чистую
  `buildLastAction` (общая для `commit()` и повтора). `stepReplayForward`
  реально применяет ход движком, `stepReplayBackward` подставляет готовый
  кадр N−1 с развёрнутой анимацией. Шаг на 1 (◀/▶/стрелки) — анимируется,
  прыжок ⏮/⏭ — мгновенный через `loadReplayFrame` (осознанно).
- **Бренд:** подпись «вариант 10×8» удалена насовсем (ключ `subtitle`,
  `.brand-sub`, проп `withSub`); знак — три восходящих пика (гребень Петуха +
  ASCENT) в `Brand.tsx`.
- **PRESETS** пополнены вариантами без инкремента (3+0/5+0/10+0/15+0);
  порядок в массиве — «без добавки» рядом со «с добавкой». zod-enum сервера
  и пикер приглашения строятся из PRESETS программно — при добавлении
  пресета больше нигде ничего менять не нужно.
- **«Как играть» (`/how-to-play`, доступна без логина):** интерактивный тур
  на базе mode `'tutorial'` в сторе (`startTutorialPractice`/
  `exitTutorialPractice`) — практика на БОЕВОЙ `Board`+`EvolutionModal`,
  реальные ходы движком. Контент — `tutorial/howToPlayLessons.ts` (позиции
  продублированы из lessons.ts + короли, т.к. работает настоящий
  legalMoves; в практике взятия слон вне зоны эволюции — иначе движок
  открывает окно формы и смешивает уроки). Тип шага `'practice'` добавлен в
  `TutStep` аддитивно; `Tutorial.tsx`/`TutorialBoard.tsx`/`lessons.ts` НЕ
  менять (в TutorialBoard только обязательный `case 'practice'` в stepDur).
  Проверка цели: подписка на стор, поле `lastMoveApplied` (ход как есть).
  Справочник эволюций — `EvolutionReference.tsx` (7 карточек, мини-доски
  5×5, движения по таблице B2). Прогресс — localStorage
  `chess2.howToPlayProgress.v1`.

## Рейтинг, ранги, матчмейкинг, лидерборд (фаза 6)

Миграции `005_rating.sql` (рейтинг/ранги) и `006_ranked_chat_invite.sql`
(пометка «рейтинговая» на карточке приглашения в чате).

- **Elo с динамическим K** — `server/lib/rating.ts` (чистые функции, без БД).
  Формула классическая; K по игроку: <10 партий → 60, 10–29 → 40, 30+ → 24,
  ≥1800 и 30+ → 16. Старт рейтинга **1000** (проставляется DEFAULT-ом в stats,
  register на него полагается). **Пол 100** — клампится в `computeRatingChange`
  по before-рейтингу (rating + delta === newRating, на полу проигрыш = 0).
  **Минимум ±1** за результативную партию (к ничьим не применяется). Округление
  модуля **один раз** при равном K (zero-sum), при разном K — каждому свой
  (лёгкая инфляция ожидаема). Опорные числа (K=24, победа): равный +12, +200 →
  +18, −200 → +6, +400 → +22, −400 → +2.
- **Рейтинг ОДИН на все контроли времени** (не по пуле/блицу/рапиду). При
  маленьком онлайне дробить пул нельзя — рейтинг никогда бы не сошёлся.
- **Защита от фарма** (`repeatMultiplier`): за 24ч с тем же соперником 1–3-я
  партии → ×1.0, 4-я → 0.5, 5-я → 0.25, 6-я+ → 0. Считается по прошлым
  завершённым рейтинговым партиям пары; cutoff в JS (без SQL-INTERVAL, ради
  pg-mem). Множитель применяется ДО округления и ДО минимума ±1.
- **8 рангов** (`src/app/lib/ranks.ts` — ЕДИНЫЙ источник правды, импортируется и
  клиентом, и, при нужде, сервером): Пешка/Оруженосец/Петух/Дозорный/Прелат/
  Таран/Феникс/Магистр ASCENT. Ранг НЕ хранится и НЕ гоняется по сети — выводится
  из рейтинга через `rankFor()`. `rankProgress()` — полоска до следующего ранга.
- **Применение при завершении** — `finishGame` (`server/sockets/game.ts`): всё в
  ОДНОЙ транзакции с идемпотентным guard-ом (`rowCount` после UPDATE status), не
  задваивает рейтинг/статистику при гонке (сдача+флажок+реконнект). Обычная
  статистика считает ВСЕ онлайн-партии; `ranked_*` — только рейтинговые. Событие
  `game-over` расширено полем `rating: { white, black } | null` (клиент берёт
  свою дельту по цвету). `is_ranked` в `games`, `*_rating_before/_delta` там же.
  Обрыв связи (`abandon`) в рейтинговой партии СПИСЫВАЕТ рейтинг (тот же путь);
  `aborted` (не состоялась) — нет.
- **Матчмейкинг** — `server/sockets/matchmaking.ts`, очередь `Map<userId,…>` **в
  памяти** (эфемерная, при рестарте теряется — клиент показывает «поиск
  прерван»). `tryMatch()` сводит глобально ближайшую по рейтингу пару без порогов
  допуска. **Критическая секция синхронна**: пара выбирается и удаляется из
  очереди БЕЗ единого `await` перед вставкой в БД — иначе игрок мог бы попасть в
  две партии. Партия создаётся `createFriendGame(..., { ranked:true, active:true })`
  — та же общая функция, что и friend/chat-invite, но сразу `active` с
  инициализированными часами. Флажок таких партий планируется в `join-game`
  (идемпотентно) — иначе таймер завёлся бы только с первым ходом.
- **Рейтинговая партия с другом** — флаг `ranked` в `friend-invite` и
  `chat:invite` (тумблер в `TimeControlPicker`); пометка «рейтинговая» в тосте
  (`InviteLayer`) и на карточке чата (`invite_ranked` в messages). Защита от
  фарма покрывает злоупотребление.
- **Лидерборд** — `server/routes/leaderboard.ts`, `GET /api/leaderboard?page=`
  (по 25). **Обязательно** `users.deleted_at IS NULL` (удалённых нет) и порог
  `ranked_games_played >= 5`. Своё место — через `COUNT(rating > моего)+1`,
  видно даже вне текущей страницы; при <5 партиях — «сыграйте ещё N».
- **Рейтинг рядом с ником везде** — `RatingBadge`/`RatingSummary`
  (`src/app/components`): PlayerBar, Friends, Chats, профили, история, тост
  приглашения, лидерборд. `rating` добавлен в ответы `/api/friends`,
  `/api/players/:username`, `/api/chat/conversations`, `/api/games/history`,
  `/api/games/:id`, `/api/stats/:id`, payload соперника в `game-state` и
  инвайтере `friend-invite`. Экраны: `/play/ranked` (RankedPage) и
  `/leaderboard` (LeaderboardPage).

## Что НЕ входит в MVP (вырезано в Pre-Code Audit §4)

- IndexedDB (только localStorage), PWA/offline, второй язык i18n.
- Расширенный FEN и PGN-экспорт (нужен только внутренний хэш позиции).
- Онлайн, ИИ-оппонент, рейтинги, Competitive-правила, сетап-драфт.
- Недостаток материала — консервативная заглушка (только очевидные случаи).

Эти пункты — НЕ реализовывать в MVP, пока пользователь не попросит явно.
