## 1. Неоднозначности и противоречия правил (GDD)

Пронумерованы как A#. Критические помечены 🔴, средние 🟡, мелкие ⚪.

**🔴 A1. «Эволюция не может давать шах в свой ход» — не формализовано.**
Главная дыра. Непонятно: запрещены ЛЮБЫЕ шахи на ходу эволюции, или только шахи, созданные НОВЫМИ возможностями формы? Пример: ладья идёт на 7-й ряд, даёт шах по вертикали базовым ходом ладьи и одновременно эволюционирует. По трактовке «любой шах» — ход нелегален (а это нормальный шахматный мотив!). По трактовке «только от новых сил» — нужно отделять источник шаха, что сложно и тоже неоднозначно. Два инженера сделают по-разному → разные легальные множества → разные партии. **Нужно одно точное правило.** (Рекомендация инженера+дизайнера: поскольку эволюции теперь сайдгрейды и шок-фактор мал, простейшее и тестируемое решение — **снять ограничение вовсе** (эволюция может давать шах). Если дизайн хочет анти-шок, тогда правило «эволюционирующий ход не может давать шах ВООБЩЕ» — тоже однозначно. Любой из двух вариантов кодируется; «по источнику силы» — нет. Это решение-блокер.)

**🔴 A2. Превращение пешки × эволюция — стэкинг не определён.**
Пешка превращается на последнем ряду (8/1) — это поле находится В зоне эволюции (дальняя треть 6–8). Вопрос: ход-превращение «заканчивается в зоне» → сработает ли ещё и E1 (например, пешка→Конь сразу же → Кавалер)? GDD это не оговаривает. **Нужно правило:** превращение и эволюция не стэкаются; свежепревращённая фигура не триггерит E1 этим же ходом (и, вероятно, считается «уже реализовавшей развитие» — не эволюционирует никогда). Иначе двойной апгрейд за один ход.

**🔴 A3. Точные движения эволюционных форм помечены «[Требует плейтеста]» — но это блокер КОДА, а не только баланса.**
Нельзя реализовать генерацию ходов, атак и проверку шаха, не зная точных движений Дозорного/Ловчего/Прелата/Ревнителя/Тарана/Опоры/Феникса. «Требует плейтеста» смешивает две разные вещи: (1) точную механику (нужна, чтобы вообще писать движок) и (2) балансовую валидацию (можно отложить). **До кода нужно зафиксировать конкретные (пусть предварительные) движения всех 7 форм**; балансовую правку чисел оставляем на плейтест.

**🔴 A4. Полное множество атак Петуха не специфицировано для детекции шаха/защиты.**
GDD задаёт ходы/взятия Петуха, но для проверки шаха и «защищённых полей» нужно явно: Петух АТАКУЕТ (а) первую фигуру по вертикали вперёд, (б) два поля по диагонали вперёд (шпора). Он НЕ атакует поля, на которые ходит вбок (это не взятие). Также: **«вперёд» для Петуха = направление стороны** (белый — к ряду 8, чёрный — к ряду 1); это критично для **превращённого в Петуха** (его «вперёд» = вперёд владельца). Без явной спецификации атак два инженера разойдутся в детекции шаха.

**🟡 A5. Бастион: точные поля назначения короля/ладьи не заданы (помечены «[плейтеста]» только про 2 vs 3 клетки).**
Чтобы реализовать рокировку, нужны конкретные поля для ОБЕИХ сторон при старте Кр f1 / Ладьи a1,j1. Балансовый «2 vs 3» — да, плейтест; но для MVP нужен дефолт. Предлагаемый дефолт (как в шахматах, король на 2 клетки): королевский фланг (к j) Кр f1→h1, Лj1→g1; ферзевый фланг (к a) Кр f1→d1, Лa1→e1. Условия: ни Кр, ни выбранная Ладья не ходили; поля между ними пусты (учесть, что между ними стоит Петух — рокировка только после его ухода); король не под/через/в шах. **Нужно зафиксировать дефолтные поля.**

**🟡 A6. Сброс счётчика 50/75-ходового правила при эволюции не определён.**
Счётчик сбрасывается на взятии/ходе пешки. Эволюция — необратимое продвижение (аналог хода пешки), но формально не пешка и не взятие. Сбрасывает ли эволюция счётчик? Не сказано. **Нужно решение** (рекомендую: да, эволюция сбрасывает счётчик — это прогресс).

**🟡 A7. Хэш позиции для троекратного повторения: что включать.**
Должен включать: расстановку (с учётом, что эволюционировавшая фигура ≠ базовой — разные типы), очередь хода, права рокировки, поле взятия на проходе. Tech Plan это упоминает, но нужно зафиксировать явно как часть определения «одинаковой позиции», иначе репетиция считается по-разному.

**🟡 A8. «Достаточный материал для мата» не определён (флаг по времени и ничья по материалу).**
С эволюционными фигурами таблица неизвестна. Для MVP это можно свести к консервативной заглушке (см. §4), но поведение при просрочке времени «у соперника недостаточно материала → ничья» требует хотя бы минимального определения. Деферрабельно при честной заглушке.

**⚪ A9. Координатная система.** Индекс = rank*10 + file; зафиксировать a1 = индекс 0 (rank0, file0), белые внизу. Мелочь, но должна быть одна.

**⚪ A10. Движение «внутри зоны» после эволюции.** Подтвердить (следует из «одноразово»): фигура, уже эволюционировавшая, повторно не эволюционирует, даже двигаясь внутри зоны. Явно записать.

**⚪ A11. Взятие на проходе для Петуха.** GDD: в MVP выключено. ОК, но записать как явное «нет», чтобы не реализовали случайно.

## 2. Что невозможно протестировать без доформализации

- **A1** (шах-на-эволюции): нельзя написать тест легальности — неизвестен ожидаемый результат.
- **A3** (движения форм): нельзя писать ни юнит-тесты, ни perft с эволюциями.
- **A4** (атаки Петуха): нельзя тестировать детекцию шаха от Петуха и маты с его участием.
- **A2** (превращение×эволюция): нет эталона для теста краевого случая.
- **A5** (поля Бастиона): нельзя написать тест рокировки без конкретных полей.
- **A6/A7**: нельзя тестировать ничьи (повторение/лимит) без точных определений.

Вывод: пока A1–A7 не закрыты, **perft и юнит-тесты движка написать нельзя** для затронутых областей — а это ядро MVP.

Это решения-формализации, НЕ пересмотр концепции. Все имеют простой дефолт.

| # | Блокер | Минимально достаточное решение (дефолт) |
|---|---|---|

| B2 (A3) | Точные движения 7 форм | Зафиксировать конкретные движения (предварительные) для Дозорный/Ловчий/Прелат/Ревнитель/Таран/Опора/Феникс. Числа баланса — плейтест. |
| B3 (A4) | Атаки Петуха + «вперёд = у владельца» | Явно: атакует первую фигуру вперёд по вертикали + 2 диагонали-вперёд; направление зависит от цвета; превращённый Петух наследует направление владельца. |

## 4. Можно отложить до плейтеста (НЕ блокеры)

- Все номиналы фигур и **балансовые числа форм** (движения зафиксированы — см. B2; тюнинг чисел потом).
- Граница зоны (фикс 6–8; альтернативы 5–8/7–8 не трогаем).
- Пат = ничья (MVP) vs победа (Competitive).
- Точное значение лимита ходов (75) и **полная** таблица «достаточного материала» — на MVP консервативная заглушка: ничья по материалу только в очевидных случаях (K vs K; K vs K+одинокий лёгкий неэволюц.), иначе доигрывается до лимита. Безопасно (риск — лишь чуть более длинные мёртвые позиции).
- Геометрия Бастиона (2 vs 3) — дефолт задан в B5.
- Взятие на проходе для Петуха (выключено).
- Тайм-пресеты под длину партий 10×8.
- Сложность обучающих пазлов, читаемость UI-значков.

## 5. Аудит технического плана

**Хватает ли архитектуры:** да. Чистое ядро + immutable state + конвейер «псевдо-легальные → фильтр шаха → терминалы» + verify-before-apply + perft — это правильный и достаточный каркас для детерминированной MVP. Разделение `engine`/`app` корректно и оставляет дверь для ИИ/онлайна.

**Переусложнение (вырезать из MVP):**
| Что | Вердикт |
|---|---|
| **IndexedDB** для сейвов | ВЫРЕЗАТЬ из MVP → только `localStorage` (autosave + продолжить). Именованные сейвы/IndexedDB — позже. |
| **PWA/offline** | Отложить на «полировку»/пост-MVP. Не нужно для первой играбельной партии. |
| **i18n RU/EN** с двумя языками | MVP — один язык, строки вынесены в один словарь; второй язык позже. |
| **Человекочитаемый расширенный FEN + PGN-экспорт** | Для MVP не нужны. Нужен только внутренний **хэш позиции** (для повторения) и сейв через стартовую позицию + список ходов (реплей). FEN/PGN — пост-MVP. |
| **Zustand** | Оставить (дёшево) ИЛИ заменить на React context для MVP — на усмотрение; не блокер. |
| **Undo/redo полным стеком** | Оставить простой стек состояний (он почти бесплатен от immutability). ОК. |

**Чего НЕ хватает в плане (добавить явно):**
- Раздел «**спецификация атак каждого типа**» (для детекции шаха) — отдельный модуль `attacks.ts` или часть `legality.ts`, чтобы Петух/формы имели единый источник атак (см. A4).
- Явный модуль/таблица движений эволюционных форм (`pieces/evolved.ts`) с конкретикой из B2.

**Что писать первым:** ядро движка в порядке §7 ниже (типы → доска → ходы фигур+Петух → apply → легальность → терминалы → perft), и только потом спецходы, эволюция, UI.

**Итог по плану:** архитектура достаточна и не раздута по сути; вырезаем 4 преждевременных пункта (IndexedDB, PWA, второй язык, FEN/PGN) → MVP быстрее до первой партии.

## 6. Roadmap разработки (5 этапов) + Definition of Done

**Этап 1 — Минимальный движок (headless).**
Типы, доска 10×8 (mailbox, a1=0), генерация псевдо-легальных ходов классических фигур + Петуха (с явными атаками per B3), `applyMove` (immutable), сериализация-хэш позиции (B7).
**DoD:** из стартовой позиции генерируются корректные псевдо-ходы; партию можно проигрывать программно ходами; **perft на глубину 1–3 совпадает с эталоном для позиции без спецходов** (эталон считается вручную/полу-автоматически и фиксируется в тестах).

**Этап 2 — Легальность ходов и терминалы.**
Атаки всех базовых типов, детекция шаха, фильтр king-safety, мат/пат; ничьи (повторение по B7, лимит по B6, заглушка материала по §4); спецходы, влияющие на легальность: Бастион (B5), взятие на проходе, превращение пешки (B4 — без стэка с эволюцией).
**DoD:** нет нелегальных ходов; корректно определяются шах/мат/пат/ничьи; **perft с рокировкой/e.p./превращением совпадает с эталоном**; набор юнит-тестов на Петуха (запрет отступления, блокировка спереди, шпора, шах от Петуха) зелёный.

**Этап 3 — Эволюция.**
Триггер зоны (6–8/1–3; Петух 7–8/1–2), 7 форм с конкретными движениями (B2) в `pieces/evolved.ts`, выбор 1 из 2 как варианты хода, ограничения (одноразовость A10, K/Q не эволюционируют, правило шаха-на-эволюции по B1), сброс счётчика по B6, статус эволюции в хэше (B7).
**DoD:** фигура эволюционирует ровно по правилам; формы двигаются/бьют/дают шах согласно B2; превращение не стэкается с эволюцией (B4); **perft с эволюциями совпадает с эталоном**; юнит-тесты на каждую форму и на правило B1 зелёные.

**Этап 4 — UI (играбельный hotseat).**
SVG-доска (10×8, подсветка зоны), фигуры/формы, подсветка легальных ходов, ввод (тап/drag), `EvolutionModal` (выбор формы), история ходов, индикаторы шаха/последнего хода, экран нового матча; autosave в localStorage + «Продолжить».
**DoD:** два человека играют полную партию на одном устройстве от старта до мата/ничьей; UI предлагает только легальные ходы; эволюция и выбор формы работают визуально; партия восстанавливается после перезагрузки. **← Первая играбельная партия.**

**Этап 5 — Плейтест-готовность.**
Таймеры (пресеты, переключение, флаг), базовые звуки/полировка по необходимости, инструмент сбора метрик (частота эволюций, длина партии, доля ничьих), свод всех `[Требует плейтеста]` и B-дефолтов в трекер для итераций баланса.
**DoD:** играбельно с часами; метрики собираются; чек-лист балансовых вопросов готов к сэлф-плею; стабильная альфа.

(Обучение/подсказки, темы/наборы фигур, экспорт-импорт, именованные сейвы — между этапами 4 и 5 или после, не блокируют первую партию.)

## 7. Порядок от первого файла до первой играбельной партии

1. `engine/types.ts` — Piece, Color, Square, Move, GameState (B8 конвенция).
2. `engine/board.ts` — mailbox 10×8, координаты, утилиты границ.
3. `engine/attacks.ts` — атакующие множества по типам (вкл. Петуха, B3).
4. `engine/pieces/classic.ts` + `engine/pieces/rooster.ts` — генерация ходов.
5. `engine/moveGen.ts` — сбор псевдо-легальных.
6. `engine/apply.ts` — reducer (immutable).
7. `engine/legality.ts` — шах, фильтр king-safety.
8. `engine/terminal.ts` — мат/пат/ничьи; `engine/hash.ts` (B7).
9. **perft + юнит-тесты зелёные → движок играбелен headless.**
10. `engine/special.ts` — Бастион (B5), e.p., превращение (B4) + тесты.
11. `engine/evolution.ts` + `engine/pieces/evolved.ts` — формы (B2), триггер, B1/B6/A10 + тесты.
12. `app/store` + `Board.tsx` + `Piece.tsx` + `MoveLayer.tsx` + `EvolutionModal.tsx`.
13. Связать ввод → `applyMove` → ререндер, hotseat, autosave. **→ ПЕРВАЯ ИГРАБЕЛЬНАЯ ПАРТИЯ.**
14. Дальше: таймеры, звук, обучение, полировка.
